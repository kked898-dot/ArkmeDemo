# 安排模块 · 设计思考

## 我理解的「安排」

传统工具把未来要做的事切分成待办、日程、提醒、任务——
但这些本质上是同一件事的不同形态：**还没发生、但需要去做的事**。

「安排」是对这些概念的重新统一。
它不强迫用户分类，不要求格式完整，
只是帮你把脑子里的事情，轻轻落下来。

---

## 反焦虑，是整个模块的设计基调

我在设计每一个细节时都在问自己：**这个设计会让用户焦虑吗？**

### 超期处理
超期的安排不显示红色，不显示感叹号，不出现"逾期"字样。
颜色变浅，文案变柔和：
- "昨天的事，还没来得及"
- "原定3天前，还没来得及"
- "放着有一阵子了，要处理一下吗"

用户看到这些文案，感受到的是理解，而不是催促。

### 「先放一放」是一个正当的状态
不是所有事情都必须完成。人生本来就有变数。
搁置不是逃避，是对现实的诚实。
搁置区默认折叠，不占主视图注意力，
底部只有一行浅色小字："还有 X 件事放着，不急"。

### 「不做了也没关系」
删除按钮没有用"删除"这个词。
放弃一件事不是失败，只是选择。

## 文案系统

产品的气质很大程度上由文案决定。

| 传统文案 | 即我的文案 |
|---------|-----------|
| 暂无安排 | 今天没有安排，好好休息 |
| 已逾期 | 原定周一，还没来得及 |
| 删除 | 不做了也没关系 |
| 完成 | 落子无悔。这项安排已被时光温柔留存 |
| 创建安排 | 有什么想记下来？ |
| 以后再说 | 先放一放 |
| 空列表 | 今天是一张白纸，想记什么就记什么 |

## AI 的两种使用方式

### 方式一：消息识别
用户在「发给自己」发送消息后，AI 自动分析是否包含安排。
识别结果按置信度分级：
- 高置信度："我注意到你提到了一件事，要记下来吗？"
- 中置信度："这条消息里好像有什么安排？你看一下"
- 低置信度：静默，不打扰用户

确认写入后，原始消息作为「关联记忆」保留在安排详情中，可追溯来源。

### 方式二：对话式创建
用户点击「+ 记一件事」→「和 AI 说说」，
进入多轮对话界面，像和朋友聊天一样描述安排，
AI 追问补全时间、地点、相关人等信息，
最终生成确认卡片，用户确认后写入。

这是「贾维斯」感的核心体现：
不是填表单，是对话。

## 数据模型的关键设计

每条安排都有 `contexts` 字段，存储所有关联的对话上下文。
这是为归集功能预留的——
当多条消息都和同一件事有关时，
它们最终会合并成一条安排，但所有来源都保留在详情里可以追溯。

`mergedFromIds` 字段同样预留，
用于记录归集时合并了哪些历史安排条目。

这不是当前版本的功能，但数据结构已经为它留好了位置。

## 还没做但想清楚了的

### 多条归集合并
爸爸、姐姐都提醒去医院 → 合并成一条安排，
详情里展示所有关联对话，用户感知到"这件事很多人都在关心"。
技术上需要相似度计算 + 用户确认的合并流程。

### AI 自动判断完成
后续对话说"我今天去医院了" → 
AI 检测到和现有安排匹配，提示用户"你好像已经做了这件事，要标记完成吗？"
这是真正的智能闭环，当前版本未实现。

### 私聊场景识别
Demo 中暂无完整的私聊好友系统，
但数据模型的 `source: 'private_chat'` 字段已预留，
产品上线后接入真实 IM 数据即可扩展。

## 版本节奏

**V1**
- 安排列表主视图，三种状态分区
- 手动创建安排
- 右滑完成 / 搁置操作
- 超期项柔和处理
