# 安排模块 · 设计思考

## 我理解的「安排」

传统工具把未来要做的事切分成待办、日程、提醒、任务……但这些其实是同一件事的不同形态：**还没发生、但需要去做的事**。

「安排」是对这些概念的重新统一。它不强迫用户分类，不要求格式完整，只是帮你把脑子里的事情轻轻落下来。

---

## 反焦虑，是整个模块的设计基调

我在设计每一个细节时都在问自己：**这个设计会让用户焦虑吗？**

**超期处理**
超期的安排不显示红色，不显示感叹号，不出现"逾期"字样。
颜色变浅，文案变柔和：
- "昨天的事，还没来得及"
- "原定3天前，还没来得及"  
- "放着有一阵子了，要处理一下吗"

用户看到这些文案，感受到的是理解，而不是催促。

**「先放一放」是一个正当的状态**
不是所有事情都必须完成。人生本来就有变数。
搁置不是逃避，是对现实的诚实。所以搁置区默认折叠，不占主视图注意力，
底部只有一行浅色小字："还有 X 件事放着，不急"。

**「不做了也没关系」**
删除按钮没有用"删除"这个词。
放弃一件事不是失败，只是选择。

## 文案系统

产品的气质很大程度上由文案决定。我把所有系统文案都过了一遍：

| 传统文案 | 即我的文案 |
|---------|-----------|
| 暂无安排 | 今天没有安排，好好休息 |
| 已逾期 | 原定周一，还没来得及 |
| 删除 | 不做了也没关系 |
| 完成 | 做到了 |
| 创建安排 | 有什么想记下来？ |
| 以后再说 | 先放一放 |
| 操作成功 | 记下来 |

这些文案都是在尝试让软件说"人话"，像一个理解你的朋友，而不是一个催你完成任务的系统。

## 版本节奏

**V1（当前）**
- 安排列表主视图，三种状态分区（待完成 / 搁置 / 已完成）
- 手动创建安排，支持时间、地点、相关人、心情、执行主体
- 右滑完成 / 搁置操作
- 超期项柔和处理
- 安排详情页，含关联对话上下文

**V2（已完成）**
- AI设置页面，支持任意服务商、自定义API地址和模型名
- 从「发给自己」的消息中AI识别安排
- 识别结果置信度分级，高置信度直接提示，低置信度更谨慎
- 确认后写入安排列表，详情中保留原始消息上下文

**下一步可以做的**
- 私聊/群聊中的安排识别
- 多条安排的归集合并
- AI自动判断安排是否已完成（根据后续对话上下文）
- 日历视图

---

## 数据模型的一个关键决策

每条安排都有 `contexts` 字段，存储所有关联的对话上下文。
这是为归集功能预留的——当多条消息都和同一件事有关时，
它们最终会合并成一条安排，但所有来源都保留在详情里可以追溯。

这不是当前版本的功能，但数据结构已经为它留好了位置。
